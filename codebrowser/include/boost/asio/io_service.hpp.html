57/// Provides core I/O functionality.
58/**
59 * The io_service class provides the core I/O functionality for users of the
60 * asynchronous I/O objects, including:
61 *
62 * @li boost::asio::ip::tcp::socket
63 * @li boost::asio::ip::tcp::acceptor
64 * @li boost::asio::ip::udp::socket
65 * @li boost::asio::deadline_timer.
66 *
67 * The io_service class also includes facilities intended for developers of
68 * custom asynchronous services.
69 *
70 * @par Thread Safety
71 * @e Distinct @e objects: Safe.@n
72 * @e Shared @e objects: Safe, with the specific exceptions of the reset() and
73 * notify_fork() functions. Calling reset() while there are unfinished run(),
74 * run_one(), poll() or poll_one() calls results in undefined behaviour. The
75 * notify_fork() function should not be called while any io_service function,
76 * or any function on an I/O object that is associated with the io_service, is
77 * being called in another thread.
78 *
79 * @par Concepts:
80 * Dispatcher.
81 *
82 * @par Synchronous and asynchronous operations
83 *
84 * Synchronous operations on I/O objects implicitly run the io_service object
85 * for an individual operation. The io_service functions run(), run_one(),
86 * poll() or poll_one() must be called for the io_service to perform
87 * asynchronous operations on behalf of a C++ program. Notification that an
88 * asynchronous operation has completed is delivered by invocation of the
89 * associated handler. Handlers are invoked only by a thread that is currently
90 * calling any overload of run(), run_one(), poll() or poll_one() for the
91 * io_service.
92 *
93 * @par Effect of exceptions thrown from handlers
94 *
95 * If an exception is thrown from a handler, the exception is allowed to
96 * propagate through the throwing thread's invocation of run(), run_one(),
97 * poll() or poll_one(). No other threads that are calling any of these
98 * functions are affected. It is then the responsibility of the application to
99 * catch the exception.
100 *
101 * After the exception has been caught, the run(), run_one(), poll() or
102 * poll_one() call may be restarted @em without the need for an intervening
103 * call to reset(). This allows the thread to rejoin the io_service object's
104 * thread pool without impacting any other threads in the pool.
105 *
106 * For example:
107 *
108 * @code
109 * boost::asio::io_service io_service;
110 * ...
111 * for (;;)
112 * {
113 * try
114 * {
115 * io_service.run();
116 * break; // run() exited normally
117 * }
118 * catch (my_exception& e)
119 * {
120 * // Deal with exception as appropriate.
121 * }
122 * }
123 * @endcode
124 *
125 * @par Stopping the io_service from running out of work
126 *
127 * Some applications may need to prevent an io_service object's run() call from
128 * returning when there is no more work to do. For example, the io_service may
129 * be being run in a background thread that is launched prior to the
130 * application's asynchronous operations. The run() call may be kept running by
131 * creating an object of type boost::asio::io_service::work:
132 *
133 * @code boost::asio::io_service io_service;
134 * boost::asio::io_service::work work(io_service);
135 * ... @endcode
136 *
137 * To effect a shutdown, the application will then need to call the io_service
138 * object's stop() member function. This will cause the io_service run() call
139 * to return as soon as possible, abandoning unfinished operations and without
140 * permitting ready handlers to be dispatched.
141 *
142 * Alternatively, if the application requires that all operations and handlers
143 * be allowed to finish normally, the work object may be explicitly destroyed.
144 *
145 * @code boost::asio::io_service io_service;
146 * auto_ptr<boost::asio::io_service::work> work(
147 * new boost::asio::io_service::work(io_service));
148 * ...
149 * work.reset(); // Allow run() to exit. @endcode
150 *
151 * @par The io_service class and I/O services
152 *
153 * Class io_service implements an extensible, type-safe, polymorphic set of I/O
154 * services, indexed by service type. An object of class io_service must be
155 * initialised before I/O objects such as sockets, resolvers and timers can be
156 * used. These I/O objects are distinguished by having constructors that accept
157 * an @c io_service& parameter.
158 *
159 * I/O services exist to manage the logical interface to the operating system on
160 * behalf of the I/O objects. In particular, there are resources that are shared
161 * across a class of I/O objects. For example, timers may be implemented in
162 * terms of a single timer queue. The I/O services manage these shared
163 * resources.
164 *
165 * Access to the services of an io_service is via three function templates,
166 * use_service(), add_service() and has_service().
167 *
168 * In a call to @c use_service<Service>(), the type argument chooses a service,
169 * making available all members of the named type. If @c Service is not present
170 * in an io_service, an object of type @c Service is created and added to the
171 * io_service. A C++ program can check if an io_service implements a
172 * particular service with the function template @c has_service<Service>().
173 *
174 * Service objects may be explicitly added to an io_service using the function
175 * template @c add_service<Service>(). If the @c Service is already present, the
176 * service_already_exists exception is thrown. If the owner of the service is
177 * not the same object as the io_service parameter, the invalid_service_owner
178 * exception is thrown.
179 *
180 * Once a service reference is obtained from an io_service object by calling
181 * use_service(), that reference remains usable as long as the owning io_service
182 * object exists.
183 *
184 * All I/O service implementations have io_service::service as a public base
185 * class. Custom I/O services may be implemented by deriving from this class and
186 * then added to an io_service using the facilities described above.
187 */

219 /// Destructor.
220 /**
221 * On destruction, the io_service performs the following sequence of
222 * operations:
223 *
224 * @li For each service object @c svc in the io_service set, in reverse order
225 * of the beginning of service object lifetime, performs
226 * @c svc->shutdown_service().
227 *
228 * @li Uninvoked handler objects that were scheduled for deferred invocation
229 * on the io_service, or any associated strand, are destroyed.
230 *
231 * @li For each service object @c svc in the io_service set, in reverse order
232 * of the beginning of service object lifetime, performs
233 * <tt>delete static_cast<io_service::service*>(svc)</tt>.
234 *
235 * @note The destruction sequence described above permits programs to
236 * simplify their resource management by using @c shared_ptr<>. Where an
237 * object's lifetime is tied to the lifetime of a connection (or some other
238 * sequence of asynchronous operations), a @c shared_ptr to the object would
239 * be bound into the handlers for all asynchronous operations associated with
240 * it. This works as follows:
241 *
242 * @li When a single connection ends, all associated asynchronous operations
243 * complete. The corresponding handler objects are destroyed, and all
244 * @c shared_ptr references to the objects are destroyed.
245 *
246 * @li To shut down the whole program, the io_service function stop() is
247 * called to terminate any run() calls as soon as possible. The io_service
248 * destructor defined above destroys all handlers, causing all @c shared_ptr
249 * references to all connection objects to be destroyed.
250 */
251 BOOST_ASIO_DECL ~io_service();

253 /// Run the io_service object's event processing loop.
254 /**
255 * The run() function blocks until all work has finished and there are no
256 * more handlers to be dispatched, or until the io_service has been stopped.
257 *
258 * Multiple threads may call the run() function to set up a pool of threads
259 * from which the io_service may execute handlers. All threads that are
260 * waiting in the pool are equivalent and the io_service may choose any one
261 * of them to invoke a handler.
262 *
263 * A normal exit from the run() function implies that the io_service object
264 * is stopped (the stopped() function returns @c true). Subsequent calls to
265 * run(), run_one(), poll() or poll_one() will return immediately unless there
266 * is a prior call to reset().
267 *
268 * @return The number of handlers that were executed.
269 *
270 * @throws boost::system::system_error Thrown on failure.
271 *
272 * @note The run() function must not be called from a thread that is currently
273 * calling one of run(), run_one(), poll() or poll_one() on the same
274 * io_service object.
275 *
276 * The poll() function may also be used to dispatch ready handlers, but
277 * without blocking.
278 */
279 BOOST_ASIO_DECL std::size_t run();

410 /// Reset the io_service in preparation for a subsequent run() invocation.
411 /**
412 * This function must be called prior to any second or later set of
413 * invocations of the run(), run_one(), poll() or poll_one() functions when a

423 /// Request the io_service to invoke the given handler.
424 /**
425 * This function is used to ask the io_service to execute the given handler.
426 *
427 * The io_service guarantees that the handler will only be called in a thread
428 * in which the run(), run_one(), poll() or poll_one() member functions is
429 * currently being invoked. The handler may be executed inside this function
430 * if the guarantee can be met.
431 *
432 * @param handler The handler to be called. The io_service will make
433 * a copy of the handler object as required. The function signature of the
434 * handler must be: @code void handler(); @endcode
435 *
436 * @note This function throws an exception only if:
437 *
438 * @li the handler's @c asio_handler_allocate function; or
439 *
440 * @li the handler's copy constructor
441 *
442 * throws an exception.
443 */
444 template <typename CompletionHandler>
445 BOOST_ASIO_INITFN_RESULT_TYPE(CompletionHandler, void ())
446 dispatch(BOOST_ASIO_MOVE_ARG(CompletionHandler) handler);
447
448 /// Request the io_service to invoke the given handler and return immediately.
449 /**
450 * This function is used to ask the io_service to execute the given handler,
451 * but without allowing the io_service to call the handler from inside this
452 * function.
453 *
454 * The io_service guarantees that the handler will only be called in a thread
455 * in which the run(), run_one(), poll() or poll_one() member functions is
456 * currently being invoked.
457 *
458 * @param handler The handler to be called. The io_service will make
459 * a copy of the handler object as required. The function signature of the
460 * handler must be: @code void handler(); @endcode
461 *
462 * @note This function throws an exception only if:
463 *
464 * @li the handler's @c asio_handler_allocate function; or
465 *
466 * @li the handler's copy constructor
467 *
468 * throws an exception.
469 */
470 template <typename CompletionHandler>
471 BOOST_ASIO_INITFN_RESULT_TYPE(CompletionHandler, void ())
472 post(BOOST_ASIO_MOVE_ARG(CompletionHandler) handler);

504 /// Fork-related event notifications.
505 enum fork_event
506 {
507 /// Notify the io_service that the process is about to fork.
508 fork_prepare,
509
510 /// Notify the io_service that the process has forked and is the parent.
511 fork_parent,
512
513 /// Notify the io_service that the process has forked and is the child.
514 fork_child
515 };
516
517 /// Notify the io_service of a fork-related event.
518 /**
519 * This function is used to inform the io_service that the process is about
520 * to fork, or has just forked. This allows the io_service, and the services
521 * it contains, to perform any necessary housekeeping to ensure correct
522 * operation following a fork.
523 *
524 * This function must not be called while any other io_service function, or
525 * any function on an I/O object associated with the io_service, is being
526 * called in another thread. It is, however, safe to call this function from
527 * within a completion handler, provided no other thread is accessing the
528 * io_service.
529 *
530 * @param event A fork-related event.
531 *
532 * @throws boost::system::system_error Thrown on failure. If the notification
533 * fails the io_service object should no longer be used and should be
534 * destroyed.
535 *
536 * @par Example
537 * The following code illustrates how to incorporate the notify_fork()
538 * function:
539 * @code my_io_service.notify_fork(boost::asio::io_service::fork_prepare);
540 * if (fork() == 0)
541 * {
542 * // This is the child process.
543 * my_io_service.notify_fork(boost::asio::io_service::fork_child);
544 * }
545 * else
546 * {
547 * // This is the parent process.
548 * my_io_service.notify_fork(boost::asio::io_service::fork_parent);
549 * } @endcode
550 *
551 * @note For each service object @c svc in the io_service set, performs
552 * <tt>svc->fork_service();</tt>. When processing the fork_prepare event,
553 * services are visited in reverse order of the beginning of service object
554 * lifetime. Otherwise, services are visited in order of the beginning of
555 * service object lifetime.
556 */
557 BOOST_ASIO_DECL void notify_fork(boost::asio::io_service::fork_event event);
558
559 /// Obtain the service object corresponding to the given type.
560 /**
561 * This function is used to locate a service object that corresponds to
562 * the given service type. If there is no existing implementation of the

564 *
565 * @param ios The io_service object that owns the service.
566 *
567 * @return The service interface implementing the specified service type.

569 */
570 template <typename Service>
571 friend Service& use_service(io_service& ios);
572
573 /// Add a service object to the io_service.
574 /**
575 * This function is used to add a service to the io_service.
576 *
577 * @param ios The io_service object that owns the service.
578 *
579 * @param svc The service object. On success, ownership of the service object
580 * is transferred to the io_service. When the io_service object is destroyed,
581 * it will destroy the service object by performing:
582 * @code delete static_cast<io_service::service*>(svc) @endcode
583 *
584 * @throws boost::asio::service_already_exists Thrown if a service of the
585 * given type is already present in the io_service.
586 *
587 * @throws boost::asio::invalid_service_owner Thrown if the service's owning
588 * io_service is not the io_service object specified by the ios parameter.
589 */
590 template <typename Service>
591 friend void add_service(io_service& ios, Service* svc);
592
593 /// Determine if an io_service contains a specified service type.
594 /**

597 *
598 * @param ios The io_service object that owns the service.
599 *
600 * @return A boolean indicating whether the io_service contains the service.
601 */
602 template <typename Service>
603 friend bool has_service(io_service& ios);

620/// Class to inform the io_service when it has work to do.
621/**
622 * The work class is used to inform the io_service when work starts and
623 * finishes. This ensures that the io_service object's run() function will not
624 * exit while work is underway, and that it does exit when there is no
625 * unfinished work remaining.
626 *
627 * The work class is copy-constructible so that it may be used as a data member
628 * in a handler class. It is not assignable.
629 */
630class io_service::work
631{
632public:
633 /// Constructor notifies the io_service that work is starting.
634 /**
635 * The constructor is used to inform the io_service that some work has begun.
636 * This ensures that the io_service object's run() function will not exit
637 * while the work is underway.
638 */
639 explicit work(boost::asio::io_service& io_service);
640
641 /// Copy constructor notifies the io_service that work is starting.
642 /**
643 * The constructor is used to inform the io_service that some work has begun.
644 * This ensures that the io_service object's run() function will not exit
645 * while the work is underway.
646 */
647 work(const work& other);
648
649 /// Destructor notifies the io_service that the work is complete.
650 /**
651 * The destructor is used to inform the io_service that some work has
652 * finished. Once the count of unfinished work reaches zero, the io_service
653 * object's run() function is permitted to exit.
654 */
655 ~work();
656
657 /// Get the io_service associated with the work.
658 boost::asio::io_service& get_io_service();
659
660private:
661 // Prevent assignment.
662 void operator=(const work& other);
663
664 // The io_service implementation.
665 detail::io_service_impl& io_service_impl_;
666};
